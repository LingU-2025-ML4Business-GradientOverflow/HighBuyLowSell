# issue5：报告任务包

## 一、任务定位

`issue5` 不是把前面结果复制粘贴到文档里。  
它负责把项目变成一条能被老师和同学理解的完整故事线。

## 二、任务目标

报告需要讲清楚：

1. 我们为什么选择这个题目
2. 我们为什么这样定义任务
3. 我们为什么这样分工
4. 我们做了哪些对比
5. 我们最后为什么选这个模型
6. 这个项目的价值和局限是什么

## 三、报告建议结构

建议报告至少包含：

1. 选题背景与价值
2. 数据来源与任务定义
3. 数据特征分析
4. 两条建模路线探索
5. 评估与模型选择
6. 结论、局限与未来改进

## 四、写作原则

1. 报告要服务于最终技术主线，不要平均描述所有尝试。
2. 报告要突出“为什么这样做”，而不只是“做了什么”。
3. 报告里的图表要能支持结论，不要只是装饰。
4. 对失败结果也可以简短说明，但不要喧宾夺主。

## 五、推荐输入材料

应主要使用：

1. `issue2` 的 EDA 结论
2. `issue3A` 和 `issue3B` 的对比结果
3. `issue4` 的最终模型选择说明

## 六、建议产出

至少需要：

1. 报告初稿
2. 图表清单
3. PPT 大纲
4. 最终结论页

## 七、与技术组的接口

`issue5` 开始前，应向 `issue4` 确认以下内容：

1. 最终主线模型是什么
2. 最终比较图表有哪些
3. 哪些结果可以写进正文
4. 哪些结果只做附录或不写

## 八、完成标准

1. 报告逻辑完整
2. 图表和文字一致
3. 结论能够自洽
4. 答辩时每位成员都能理解主线

## 九、风险提醒

1. 不要在报告阶段重新定义项目主线。
2. 不要把所有实验结果平铺直叙地堆进去。
3. 不要只写结果，不写为什么这样做。
4. 不要等到最后一天才开始整理图和表。
