## 二、建议

### 1. 建议先把当前版本定位为“候选特征池”，不要直接默认全量特征都进入模型

原因：

1. 当前特征数量已经明显增加，已经足够支撑后续探索。
2. 如果后面直接全量进入模型，不利于判断到底是哪些特征真正有用。
3. 对课程项目来说，特征收敛和解释清楚，通常比继续无限扩展更重要。

其他重要信息：

建议后续区分：

1. 核心特征集
2. 扩展技术特征集

这样后面做 `issue3A` 时更容易比较。

### 2. 建议修正目标列最后一行的处理方式

原因：

1. 如果直接使用 `shift(-1) > close` 再转为 `int`，每个 symbol 的最后一行可能会被错误标成 `0`。
2. 这会引入错误标签，虽然比例不大，但属于可以避免的基础问题。

其他重要信息：

更稳妥的方式是：

1. 先生成未来一天价格列
2. 对未来价格为空的行直接删除
3. 再生成目标列

### 3. 建议减少明显冗余特征，特别是在第一版实验中

原因：

1. 一些技术指标本身高度相关。
2. 冗余特征过多会让模型解释变差。
3. 对 `Logistic Regression` 这类 baseline，过高相关性不一定带来真实提升。

其他重要信息：

后续更建议保留“代表性衍生特征”，而不是把所有中间计算量都直接送入模型。

### 4. 建议为后续 pooled 方向提前考虑“相对化特征”

原因：

1. 多股票联合建模时，绝对价格水平容易让模型学到“这是哪只股票”，而不是学到更通用的行为信号。
2. 如果以后做 `issue3B`，相对化特征会更适合比较。

其他重要信息：

例如可以考虑：

1. `close / sma_n - 1`
2. `close / ema_n - 1`
3. `bb_width`
4. `bb_position`

### 5. 建议尽快统一输出接口

原因：

1. `issue3A`、`issue4`、报告阶段都需要统一输入格式。
2. 如果目标列名、输出文件名、保留字段各不相同，后面会反复做无意义适配。

其他重要信息：

建议统一确认：

1. 目标列名
2. 输出文件路径
3. 元信息字段是否保留 `date/symbol/close`
