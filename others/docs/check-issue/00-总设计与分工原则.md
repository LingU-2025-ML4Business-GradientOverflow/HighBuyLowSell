# 总设计与分工原则

## 一、为什么要这样分工

我们当前时间紧，且团队成员的工程协同经验不完全一致。  
如果继续用“大家分别做一点模型、最后再合并”的方式，最容易出现的结果是：

1. 数据口径不一致，结果不可比。
2. 特征和标签被不同成员各自修改，后续无法判断哪个结果可信。
3. 每个人的代码都只能在自己电脑上跑，最后无法集成。
4. 报告只能写成“做了很多事”，但讲不清技术判断过程。

所以现在采用的不是“平均分工”，而是“决策链分工”。

## 二、这套分工的核心逻辑

当前项目按决策链分成 4 段：

1. `issue2`
目标：统一数据理解、标签定义和特征口径。

2. `issue3A`
目标：沿单股建模方向探索，判断单股是否更稳定、更有解释性。

3. `issue3B`
目标：沿多标的联合建模方向探索，判断 pooled 建模是否更有泛化价值。

4. `issue4`
目标：把探索结果放到同一评价体系下比较，做模型选择并形成最终技术主线。

5. `issue5`
目标：把最终技术主线转成课程报告和答辩表达。

## 三、为什么不让所有人都去做模型

因为现在项目最大风险不是“模型太简单”，而是“没有统一技术主线”。

如果人人都直接做模型，会出现：

1. 同样的特征做了多遍。
2. 同样的数据切分做了多遍。
3. 最后只剩下一堆数值，却无法说明为什么选这个模型。

所以必须先由 `issue2` 把地基打平，再让 `issue3A/3B` 去做真正有信息量的探索。

## 四、为什么 issue3 要分成 A 和 B 两条线

因为 `issue3` 的本质不是重复训练，而是比较两条有代表性的技术路线。

推荐两条路线如下：

1. `issue3A`：单股建模路线
- 重点看单个标的的时序行为是否更稳定。
- 适合回答：“单股模型是否比 pooled 更容易得到清晰结论？”

2. `issue3B`：多标的联合建模路线
- 重点看多个股票一起训练时，是否能通过共享模式得到更好的泛化。
- 适合回答：“不同股票放在一起训练是否值得？”

这样分，最后才能形成真正可比较的结论，而不是两个同学做了高度重复的东西。

## 五、为什么 issue4 不只是“回测”或“评估”

`issue4` 实际上是整个项目的收敛层。

它要回答的不是“哪个数高一点”，而是：

1. 哪条路线更值得写进最终报告？
2. 哪个模型结果更可信？
3. 哪个结果只是行情窗口运气，哪个结果更有方法意义？
4. 哪个结果能在答辩时讲清楚？

所以 `issue4` 负责的是：

- 统一评价标准
- 统一比较框架
- 最终模型选择
- 最终叙事主线收敛

## 六、为什么 issue5 单独拆出来

因为报告不是把代码截图贴进去。  
报告需要做的是把“问题 -> 方法 -> 结果 -> 结论”讲成一条线。

如果没有单独的 `issue5`，最后会出现：

1. 技术结果有，但没人整理成老师能看懂的表达。
2. 图和表很多，但前后逻辑不通。
3. 模型选择理由说不清楚。

因此 `issue5` 必须在 `issue4` 的选择结果基础上进行，而不是提前写死。

## 七、协作原则

1. `issue2` 是统一口径，不是个人自由发挥。
2. `issue3A/3B` 是有边界的探索，不是无限扩模型。
3. `issue4` 是做选择，不是重新造一套新模型。
4. `issue5` 是表达收敛，不是重新定义技术路线。

## 八、如果不这样做，会有什么风险

1. 数据和标签不统一，后面的模型结果无法比较。
2. 两个建模方向看起来不同，实际上只是做了两遍重复劳动。
3. 最后做评估时，没有统一输入格式，无法比较。
4. 报告阶段发现没有明确主线，只能临时拼接。
5. 时间被消耗在协调混乱上，而不是技术推进上。

## 九、这样做的好处

1. 每个人都知道自己工作的输入和输出是什么。
2. 探索是有方向的，不是发散的。
3. 最后模型选择是有依据的，不是拍脑袋。
4. 报告和答辩时，可以讲出清晰的技术判断过程。
5. 即使最后结果不是很强，也能体现出方法论成熟度。

## 十、项目总接口

可以把项目理解成下面这条链：

1. `issue1` 交付干净的数据入口
2. `issue2` 交付统一的数据理解和特征口径
3. `issue3A/3B` 交付两条可比较的实验路线
4. `issue4` 交付最终模型选择和评价结论
5. `issue5` 交付报告和答辩表达

这条链一旦清楚，协作就不容易误解。
